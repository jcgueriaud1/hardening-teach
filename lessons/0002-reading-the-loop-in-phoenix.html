LESSON 02 Coding-agent harnesses · ~8 min · observing the loop

Reading the loop in your Phoenix traces

In Lesson 1 the agentic loop was a mental model. In Phoenix it's a picture you can point at. This lesson turns your labeled span tree into a diagnosis.

The win for this lesson Given a Phoenix trace of Claude Code working on your Vaadin app, you can find the AGENT → LLM / TOOL structure, name which loop step each span belongs to, and point at the exact span where a Flow mistake happened — in under a minute.

Your spans are the loop, relabeled

You confirmed your spans arrive labeled AGENT, LLM, and TOOL. That labeling is OpenInference — it means an enrichment layer (the Arize hooks plugin or OTLP bridge) has mapped Claude Code's native span tree onto standard span kinds1:

  • claude_code.interaction → AGENT — one whole turn of the loop
  • claude_code.llm_request → LLM — one call to Claude (a decision)
  • claude_code.tool → TOOL — one tool call (an observation or action)

So Lesson 1's gather → act → verify isn't hidden inside the model — it's laid out as a tree you can read. Here is the same "add a delete button to the user grid" turn, as it looks in Phoenix:

▸ AGENT  claude_code.interaction   "Add a delete button to the user grid"
   ├─ LLM   llm_request             decide what to do
   ├─ TOOL  search_vaadin_docs      GATHER · grounds the real Grid API      ← step 1
   ├─ LLM   llm_request             write the edit
   ├─ TOOL  Edit  UserView.java     ACT · applies the change                ← step 2
   ├─ TOOL  Bash  mvn -q compile    VERIFY · status = ERROR            ← step 3
   ├─ LLM   llm_request             read the error, fix
   └─ TOOL  Edit  UserView.java     ACT · corrected                         ↺ repeat

Read a trace top-down and you are watching the agent think. The AGENT span is the turn; every TOOL child is a moment where it touched your project; every LLM child is a decision that produced the next action.2

How to read a trace in five moves

  1. Find the AGENT root. That's one turn / one prompt. Everything nests under it.
  2. Scan the TOOL children for the three loop moments — a gather (search/read), an act (Edit/Write), a verify (Bash build/test). Missing one is a smell, not just a gap.
  3. Check TOOL span status. A red/ERROR status on a build or test span is your verify step failing — the highest-value thing to look at first.
  4. Open the LLM span just before a bad TOOL. Its input/output shows the reasoning and the tool arguments — why the agent did the wrong thing. (Content is redacted unless prompt/tool logging is enabled.3)
  5. Count the children. A one-line change that took 15 LLM + 20 TOOL spans is low path convergence — correct, maybe, but inefficient.4

Flow-specific smells to hunt for

Smell 1 · no gather before code An LLM span goes straight to an Edit TOOL with no search_vaadin_docs (or file-read) TOOL before it. The model is coding your Grid/Binder from memory — the classic route to hallucinated or pre-25 Vaadin API. Fix lives in step 1 (grounding).
Smell 2 · raw HTML in the edit An Edit TOOL whose diff adds a raw <div>/<button> instead of a Vaadin Button/Grid component. Visible in the tool's arguments if tool-content logging is on.
Smell 3 · verify missing or ignored No Bash compile/test TOOL at all, or one with ERROR status that the AGENT turn ends right after. The agent declared victory without checking — models over-trust their own work.5 This is where a build/TestBench gate earns its keep.
Why this matters for the mission Every fix you'll make to "harden Vaadin for agents" shows up here as a changed trace: turn on the Vaadin MCP server and a search_vaadin_docs TOOL appears in step 1; add a verify skill and a green build TOOL appears in step 3. Phoenix is how you'll prove a change helped.

Check yourself (recall before revealing)

1. From memory: which span kind marks one whole turn of the loop, and which marks a single tool call?
Reveal AGENT wraps the whole turn; TOOL is one tool call. LLM spans are the decisions in between.
2. In one AGENT turn you see an LLM span go straight to an Edit TOOL, with no doc-lookup or file-read TOOL before it. Which loop step is at risk?
  • A) Gathering the context
  • B) Verifying the result
  • C) Taking the action
  • D) Naming the button
Reveal A. No gather span means the model is writing Vaadin code from memory — ungrounded step 1, the hallucinated-API route. The act (C) happened fine; verify (B) comes later.
3. One AGENT turn contains 15 LLM and 20 TOOL spans for a one-line change that ends up correct. Which quality dimension is poor?
  • A) Efficiency — path convergence
  • B) Correctness — right answer
  • C) Communication — clear reply
  • D) Reliability — valid tool args
Reveal A. It got there (correct), but wandered — many more steps than the task needs is low path convergence, an efficiency problem. That's a distinct axis from whether the answer was right.
Beta note: if you ever see a turn with only LLM spans and no AGENT/TOOL tree, that's the known Agent-SDK-streaming limitation — the full tree emits from the claude -p path. You're seeing AGENT/TOOL, so you're fine.6
