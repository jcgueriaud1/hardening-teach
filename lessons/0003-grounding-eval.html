LESSON 03 Coding-agent harnesses · ~9 min · from reading to measuring

Your first eval: is the agent grounded?

Reading one trace tells you about one run. An eval answers the same question about every run, automatically — so you can prove a change helped instead of hoping.

The win for this lesson You can define one eval — "did the agent consult the Vaadin docs before writing Vaadin code?" — compute it across all your traces as a single grounded-rate, and use that number to prove whether a CLAUDE.md rule actually changed the agent's behavior.

Why an eval, not another screenshot

Your trace left a real question open: the Vaadin MCP server is enabled, but in the turn we could see, no search_vaadin_docs TOOL ran before the Writes. Enabled ≠ used — a tool's presence doesn't earn its calls; the agent will code from memory unless nudged.1 So: how often does your agent actually ground itself? You cannot answer that by eyeballing traces one at a time. That is what an eval is for — a repeatable judgment applied to every trace.

Two kinds of evaluator — pick the cheap one when you can

Phoenix supports two2:

Code-based evaluator Deterministic checks over the trace's structure or attributes — "did span X precede span Y?", "what was the build's exit status?", "how many tool calls?". Cheap, fast, exact.
LLM-as-a-judge An LLM scores something subjective — "is this code idiomatic Vaadin?", "was the answer correct?". In Phoenix it uses function calling to return a structured label and an explanation by default.3

"Did a doc-search precede the first edit?" is a structural fact — so it's a code-based evaluator. Reach for an LLM judge only when the question needs judgment. Save the model (and the money) for questions code can't answer.

The grounding eval, concretely

Label each AGENT turn one of three ways — and the third one is the point:

  • grounded — a Vaadin-doc lookup happened before the turn's first Vaadin write
  • not_grounded — a Vaadin write with no preceding doc lookup
  • no_vaadin_code — the turn wrote no Vaadin code, so it is excluded from the rate

A turn that only edits YAML or backend logic has no Vaadin API to get wrong — counting it as "ungrounded" would poison the metric. So the rate is grounded / (grounded + not_grounded), measured only where grounding actually applies. This is the same "measure the right thing" discipline Lesson 5 will insist on for the judge. As an algorithm over spans:

// npm install @arizeai/phoenix-client   (TS client uses PHOENIX_HOST, not PHOENIX_COLLECTOR_ENDPOINT)
import { createClient } from "@arizeai/phoenix-client";
import { getSpans } from "@arizeai/phoenix-client/spans";

const client = createClient();                       // reads PHOENIX_HOST / PHOENIX_API_KEY
const { spans } = await getSpans({ client, project: { projectName: "expense-manager" }, limit: 1000 });

function classifyTurn(turn) {                          // -> "grounded" | "not_grounded" | "no_vaadin_code"
  turn.sort((a, b) => +new Date(a.start_time) - +new Date(b.start_time));
  let saw = false;
  for (const s of turn) {
    const name = (s.name ?? "").toLowerCase();
    if (name.includes("search_vaadin_docs") || name.includes("get_full_document") || s.span_kind === "RETRIEVER") saw = true;
    // a write counts only if it touches Vaadin (needs tool-content logging in the span attrs)
    const isVaadinWrite = (name.startsWith("write") || name.startsWith("edit"))
      && JSON.stringify(s.attributes ?? {}).toLowerCase().includes("com.vaadin");
    if (isVaadinWrite) return saw ? "grounded" : "not_grounded";   // first Vaadin write decides
  }
  return "no_vaadin_code";                             // wrote no Vaadin code -> excluded from the rate
}

// each turn is its own trace
const byTrace = new Map();
for (const s of spans) {
  const t = s.context.trace_id;
  if (!byTrace.has(t)) byTrace.set(t, []);
  byTrace.get(t).push(s);
}
const labels = [...byTrace.values()].map(classifyTurn);
const vaadin = labels.filter((l) => l !== "no_vaadin_code");        // only where grounding applies
const rate = vaadin.filter((l) => l === "grounded").length / vaadin.length;
console.log(`grounded-rate: ${(rate * 100).toFixed(0)}%  (over ${vaadin.length} Vaadin-writing turns)`);

How does the eval know a write "touches Vaadin"? It scans the write span's attributes for a signal like com.vaadin — which requires tool-content logging to be on. Run grounding_eval.ts --inspect to see exactly what your write spans capture, then tune the signal.

The full, runnable version — with a discovery step that prints your real span names first, and optional log-back — is grounding_eval.ts (run with npx tsx grounding_eval.ts --project expense-manager --discover). It labels each turn three ways (grounded / not_grounded / no_vaadin_code) and reports the rate over Vaadin-writing turns only. Run --discover to confirm your span names (MCP tools may appear namespaced, e.g. mcp__vaadin__search_vaadin_docs) and --inspect to see what your write spans capture, then tune the matchers and the Vaadin-write signal.

To write the verdict back into Phoenix so it shows on every trace and trends over time, log it as an annotation (Phoenix renamed evaluations → annotations2): addSpanAnnotation({ client, spanAnnotation: { spanId, name: "grounded", annotatorKind: "CODE", label, score } }) from @arizeai/phoenix-client/spans. Annotations attach to a span, so pass the turn's root AGENT span's context.span_id.

The feedback loop this unlocks 1. Measure today's grounded-rate (say 40%). 2. Add a CLAUDE.md rule: "Before writing or changing a Vaadin component, search the Vaadin docs."4 3. Re-run the same tasks and re-measure. If the rate jumps to 85%, you've proven the rule works — in numbers, not vibes. That is the entire mission in one loop.
Rigorous version: pin a fixed set of tasks as a Phoenix dataset and run each change as an experiment, so before/after comparisons are apples-to-apples on versioned inputs.5 Same idea, controlled inputs.

Check yourself (recall before revealing)

1. From memory: what separates a code-based evaluator from an LLM-as-a-judge, and which fits "did a doc-search precede the first edit?"
Reveal Code-based checks objective/structural facts deterministically; LLM-as-a-judge scores subjective/semantic quality. "Did a doc-search precede the edit?" is structural → code-based.
2. Which of these should be an LLM-as-a-judge rather than a code check?
  • A) "Did a doc-search precede the edit?"
  • B) "Is the generated code idiomatic Vaadin?"
  • C) "How many tool calls ran this turn?"
  • D) "Did the build step exit with zero?"
Reveal B. "Idiomatic" is a judgment — no deterministic rule captures it, so it needs an LLM judge. A, C, and D are all structural facts a few lines of code decide exactly.
3. You add the "search docs first" rule and the grounded-rate goes 40% → 85%. What have you demonstrated?
  • A) The change improved grounding, measurably
  • B) The generated code is now bug-free
  • C) The agent became faster at everything
  • D) The per-run cost is guaranteed lower
Reveal A. The eval measures grounding and only grounding — it went up, so that behavior improved. Correctness (B), speed (C), and cost (D) are separate evals; don't infer them from this one.
