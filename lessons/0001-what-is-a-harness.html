LESSON 01 Coding-agent harnesses · ~7 min · the core loop

What a harness actually is

One prompt does not become one answer. It becomes a loop. Once you can see the loop, every Vaadin failure and every Phoenix trace has an address.

The win for this lesson By the end you can explain, from memory, the four-step loop a harness runs — and point to exactly which step a hallucinated Vaadin API, a raw-<div> mistake, and an untested change each live in. That address is what makes "agent-readiness" a concrete engineering task rather than a vibe.

The harness is the scaffolding, not the model

It's tempting to think "Claude wrote this code." More precisely: a model proposed actions, and a harness executed them. The harness is the program around the model that assembles context, exposes tools, calls the model, runs the model's tool calls, feeds the results back, and manages context across turns. Anthropic describes the Claude Agent SDK — the same machinery that powers Claude Code — as exactly this: "a general-purpose agent harness."1

The distinction that unlocks everything: a workflow runs models and tools through predefined code paths; an agent is a model that dynamically directs its own process and tool use.2 Claude Code is an agent. You don't script its steps — you shape the environment it decides within. That is what you're hardening when you make Vaadin agent-ready.

The loop: gather → act → verify → repeat

Anthropic's canonical description of what an agent does, each turn, is four moves:1

1. Gather context
read files, search, MCP docs → 2. Take action
call a tool (edit, run, query) → 3. Verify work
tests, build, visual check ↺ repeat
until the task is done

The model isn't loaded with your whole project. It works inside a context window — a finite "attention budget" (200K tokens in Claude Code) that degrades as it fills.3 So step 1 is not free: the agent must choose what to pull in, ideally just-in-time — fetching what it needs now rather than pre-loading everything.3 Every design decision you'll make later (an MCP server, a CLAUDE.md, a skill) is really a decision about what lands in that window, at which step, in the loop.

Watch the loop run on a real Vaadin prompt

You type: "Add a delete button to the user grid." Here is the loop, annotated with where Vaadin projects break5 and how each moment shows up as a span in your Phoenix trace6:

1 · Gather context The agent reads your grid class and needs to know the current Grid API. Failure mode: with no grounding it recalls a pre-25 or invented Vaadin API — the single most common agentic-coding complaint on the Vaadin forum. Fix's home: this step — the Vaadin MCP server injects real, version-correct docs here. In Phoenix: a RETRIEVER or TOOL span (the doc lookup), feeding an LLM span.
2 · Take action The agent edits the file. Failure mode: it hand-rolls a raw <div> button instead of Vaadin's Button/Grid component API. Fix's home: also upstream in step 1 (better component knowledge) — but the mistake surfaces here. In Phoenix: a TOOL span (the file edit), nested under an AGENT span.
3 · Verify work Does it compile? Does the button render? Failure mode: the agent declares victory without running anything — models are notoriously poor at grading their own output.4 Fix's home: give verification a real interface — a build, a test, a Playwright visual check (the visual-verification skill in Vaadin's reference repo).7 In Phoenix: a TOOL span running the build/test, whose status tells you pass or fail.

Notice the payoff: "make Vaadin agent-ready" decomposes into "put the right thing in the right loop step." Grounding docs → step 1. Component skills → step 1. A verify harness → step 3. You are not improving "the AI" in the abstract; you are engineering three specific moments.

Check yourself (retrieval, not recognition)

Answer before opening each toggle. Struggling to recall is the point — it's what builds durable memory, not fluency that fades.

1. From memory: what are the four steps of the agentic loop, in order?
Reveal Gather context → take action → verify work → repeat. If you got three of four, the one people drop is usually verify — which is exactly the step most agent failures hide in.
2. An agent writes a raw <div> grid instead of Vaadin's Grid, and invents an API that doesn't exist. Which upfront addition most directly prevents this?
  • A) A bigger context window
  • B) The Vaadin MCP server
  • C) A second worker subagent
  • D) A longer system prompt
Reveal B. The failure is bad context at step 1 — the model is guessing at Vaadin's API. The MCP server grounds that step with real, version-correct docs. A bigger window (A) gives it more room to guess wrong; more subagents (C) and a longer prompt (D) don't supply the missing facts.
3. In a Phoenix trace, which span kind wraps a group of LLM and TOOL calls?
  • A) CHAIN
  • B) AGENT
  • C) LLM
  • D) TOOL
Reveal B — AGENT. An AGENT span "encompasses calls to LLMs and Tools." CHAIN is glue between steps; LLM and TOOL are the individual calls it contains.
