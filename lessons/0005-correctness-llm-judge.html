LESSON 05 Coding-agent harnesses · ~9 min · the judgment axis

Correctness: an LLM-judge for real Vaadin

Code can check whether the agent grounded itself. It can't check whether the result is idiomatic Vaadin 25 or confident nonsense. That needs judgment — and a judge you can trust.

The win for this lesson You can build an LLM-as-a-judge that labels whether the agent's Vaadin code uses real, idiomatic Vaadin 25 API — and, crucially, you know the two things that make its verdict trustworthy: ground it in real docs, and validate it against your own labels before believing a number.

Why this one can't be code

Lesson 3's rule: use the cheap deterministic check when the question is structural. "Did a doc-search precede the edit?" — code. "How many tokens?" — code. But "is this idiomatic Vaadin 25, or a hallucinated API?" has no deterministic rule. It needs world knowledge and judgment, so it's an LLM-as-a-judge: an LLM scores the output and, in Phoenix, returns a structured label plus an explanation via function calling.1

The trap: your judge has the coder's disease

The core insight of this lesson An LLM judge asked "is this real Vaadin API?" can hallucinate exactly like the coder did. Ask it from memory and it will confidently bless invented methods and flag real ones — you'll have automated a second unreliable opinion. The judge needs the same fix the coder needed: grounding.

So feed the judge the real Vaadin docs for the components in the diff — retrieved just-in-time from the Vaadin MCP server (get_full_document)2 — and instruct it to judge only against that context, never its own memory. This is the whole course rhyming: the same MCP server that should ground your coder (Lesson 3) also grounds your evaluator. Ground once, use twice.

The judge, concretely

Give it a tight rubric and a structured output — a category, the offending symbols, and an explanation (Phoenix evals return explanations by default, which is also how you debug the judge).1

const rubric = `You are grading a code diff that uses Vaadin Flow (Java).
Judge ONLY against the official Vaadin docs provided below. Do NOT rely on memory.
Return one of:
  idiomatic       - uses real Vaadin 25 API the recommended way
  non_idiomatic   - real API, but misused (e.g. raw layout where a component fits)
  hallucinated    - calls a class/method that does not exist in the provided docs
List the exact offending symbols. Explain in one sentence.

--- OFFICIAL VAADIN DOCS (retrieved) ---
${vaadinDocs}
--- CODE DIFF ---
${codeDiff}`;

// 1. from the trace: the code the agent wrote (the Edit/Write TOOL span args)
// 2. retrieve real docs for the components mentioned (Vaadin MCP get_full_document)
// 3. run the judge with structured (tool-call) output -> { label, symbols[], explanation }
// 4. log the verdict back to Phoenix as an annotation (as in Lessons 3-4)

Don't trust it until it agrees with you

Validate the judge (a meta-eval) Phoenix's built-in eval templates are trustworthy because they're benchmark-tested.3 Your custom judge isn't — yet. Hand-label ~20 real diffs from your traces (idiomatic / non_idiomatic / hallucinated), run the judge over the same 20, and measure agreement. If it disagrees with you, fix the rubric or the grounding before you believe a single dashboard number. An unvalidated judge is a confident stranger.

Three axes make a scorecard

You now have the few metrics that actually indicate Vaadin agent quality4, each measured the right way:

  • Grounding (Lesson 3) — code-based: did it consult the docs?
  • Efficiency (Lesson 4) — code-based: tokens/steps, not duration.
  • Correctness (this lesson) — LLM-judge, grounded & validated: is it real, idiomatic Vaadin?

Mission success is these three moving together: turn on grounding and correctness should rise; scope tasks down and efficiency should rise — and Phoenix shows all three, before and after.

Check yourself (recall before revealing)

1. From memory: why must an LLM judge for "idiomatic Vaadin" be grounded, when a code check doesn't need to be?
Reveal The judge shares the coder's weakness — it can hallucinate Vaadin API from memory, so it must judge against retrieved real docs to be trustworthy. A code check tests a structural fact that needs no world knowledge, so it can't hallucinate.
2. What is the biggest risk of an ungrounded Vaadin LLM-judge?
  • A) It blesses hallucinated API confidently
  • B) It runs a little slower each call
  • C) It costs a few cents extra
  • D) It returns no explanation text
Reveal A. It reproduces the coder's hallucination — approving invented API and flagging real API — so you've automated a second unreliable opinion. Speed and cost (B, C) are minor; explanations (D) it still returns.
3. Before you trust the judge's numbers on a dashboard, you should first…
  • A) Validate it against your own labels
  • B) Deploy it to production right away
  • C) Raise its sampling temperature high
  • D) Remove its explanation field first
Reveal A. Hand-label a small set and check the judge agrees before believing it. An unvalidated judge might be systematically wrong — and confidently so.
