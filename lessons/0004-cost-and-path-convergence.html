LESSON 04 Coding-agent harnesses · ~9 min · efficiency you can measure

Cost & path convergence: the expensive turn

One of your turns spent 18M tokens and $16 on a single prompt. Correct isn't enough — this lesson makes efficiency a number you can flag, explain, and fix.

The win for this lesson You can flag cost-outlier turns automatically, read inside one to see why it blew up (context thrash vs. legitimately big), and name the levers that shrink it — so "Turn 2 cost $16" becomes a bug you can close, not a mystery.

Measure effort honestly — three signals, and one trap

Last lesson's laptop-sleep turn taught the trap: duration is wall-clock and includes idle time, so it is a lie about effort. The honest signals are:

  • Tokens per turn — the real driver of cost and of context pressure.
  • Tool-call / span count per turn — how many steps the agent took.
  • Path convergence — minimum steps for this task type ÷ steps actually taken, a 0–1 score. 1.0 = optimal; low = it wandered.1

Path convergence is the sharpest, but it needs ground truth: you must know the minimum steps a task type should take. Until you have that, token/span outlier detection is the pragmatic first eval — no ground truth required.

The cost-outlier eval, concretely

Sum each turn's tokens, compare to the median, flag the turns that dwarf it:

# pip install arize-phoenix-client pandas   (legacy px.Client() was removed in Phoenix v14)
from phoenix.client import Client
spans = Client().spans.get_spans_dataframe(project_identifier="expense-manager", limit=5000)

# tokens per AGENT turn (group by trace). Note the "attributes." prefix, and that
# token columns are NaN on non-LLM spans -> fill before summing.
tok   = (spans["attributes.llm.token_count.total"].fillna(0)
         .groupby(spans["context.trace_id"]).sum())
steps = spans[spans["span_kind"] == "TOOL"].groupby("context.trace_id").size()

median = tok.median()
outliers = tok[tok > 5 * median].sort_values(ascending=False)   # >5x median = investigate
print(outliers)          # your Turn 2 (18M) lands at the top
print(steps.reindex(outliers.index))   # ...and how many tool steps it took

Log the flag back into Phoenix as an annotation (same mechanic as Lesson 3 — Client().spans.log_span_annotations_dataframe(...)), and every future run self-reports its outliers.2 No more scrolling the sessions list by eye.

Now read inside Turn 2 — why 18M?

A flag says which turn; the spans say why. The context window is a finite attention budget with diminishing returns as it fills3 — 18M tokens in one turn means it was refilled many times over. Look for the thrash signature:

Signature 1 · re-reading the same file Multiple Read TOOL spans on the same large file across the turn. The agent keeps re-pulling context it already had — pure waste.
Signature 2 · ballooning LLM input Successive llm_request spans whose prompt token count climbs steeply. The window is filling with history and tool output faster than it's being compacted.
Signature 3 · giant tool outputs A Bash/Read TOOL that dumped a huge output (a full build log, a whole generated file) straight into context, which every later step then re-pays for.

The root cause is usually one named failure mode: the agent over-scoped — tried to do too much in one turn, filled the window, and thrashed.4 "Implement #23" end-to-end in a single turn is exactly the kind of ask that triggers it.

The levers — and how each shows up in the trace

Shrink the expensive turn 1. Scope smaller. Decompose "implement the issue" into narrower steps → each turn stays inside the window. 2. Delegate exploration to a subagent — it burns its own context and returns a ~1–2k-token summary, keeping the lead agent's window lean.5 3. Keep big/irrelevant files out of reach (ignore build output, generated code) so a Read can't dump megabytes. 4. Trust compaction instead of re-reading — don't force the agent to reconstruct context by hand.

Mission tie: each lever is a change you can prove. Measure median tokens/turn → apply a scoping rule in CLAUDE.md or add a research subagent → re-measure. If Turn-2-shaped outliers disappear and median cost drops, the agent-readiness change worked — in dollars.

Check yourself (recall before revealing)

1. From memory: why is duration a bad efficiency signal, and what are two honest ones?
Reveal Duration is wall-clock — it includes idle time (sleep, waiting on prompts). Honest signals: tokens per turn and tool-call/span count (and path convergence, if you know the minimum steps).
2. Which pattern inside a turn best explains an 18M-token cost?
  • A) Re-reading the same files repeatedly
  • B) Making a single small file edit
  • C) Doing one quick documentation lookup
  • D) Emitting one short final response
Reveal A. Re-pulling context it already had refills the window over and over — the thrash signature. B, C, and D are all small, bounded token costs.
3. Which lever most directly shrinks a turn's context without dropping capability?
  • A) Delegating exploration to a subagent
  • B) Raising the sampling temperature
  • C) Disabling the verification step
  • D) Increasing the max output tokens
Reveal A. A subagent explores in its own window and returns a small summary, so the lead agent's context stays lean. B changes randomness, C removes a safety net, D grows context — none reduce it safely.
