REFERENCE Canonical vocabulary

Glossary

The shared language for this course. When a term below appears in a lesson, it means exactly this. Opinionated on purpose: where the field uses several words for one idea, one is chosen and the rest are listed as avoid aliases.

How this grows This is a living document. Terms are added or sharpened as your understanding deepens — a definition written today may be revised in week six. That revision is the learning.

Harness & the agent loop

Harness
The scaffolding around a model that runs the agent: it assembles context, exposes tools, calls the model, executes the model's tool calls, feeds the results back, and manages context across turns. Anthropic calls the Claude Agent SDK "a general-purpose agent harness." Avoid: wrapper, framework, the AI. [Agent SDK]

Agentic loop
The repeating cycle the harness drives: gather context → take action → verify work → repeat, until the task is done. The model chooses each action; the harness executes it and returns the result. Avoid: the algorithm, the pipeline. [Agent SDK]

Agent vs. workflow
A workflow orchestrates models and tools through predefined code paths. An agent is a model that dynamically directs its own process and tool use. Claude Code is an agent; a fixed prompt-chain is a workflow. [Building Effective Agents]

Turn
One pass through the loop: the model is invoked once, may emit text and/or tool calls, and the harness runs those calls. A task usually takes many turns.

Verification
The step where the agent (or the harness) checks the work before continuing — tests, build output, a rubric, a visual diff. It "finishes" the loop and stops errors compounding. Models are poor at grading their own work, so external, rules-based verification is preferred. [Harnesses]

Context

Context window
The finite set of tokens the model can attend to in one inference pass — a scarce "attention budget" with diminishing returns as it fills. Claude Code's default is 200K tokens. Avoid: memory, RAM. [Context Engineering, CC docs]

Context engineering
The strategies for curating and maintaining the optimal set of tokens in the window over time (system prompt, tools, history, retrieved data). The "natural progression of prompt engineering": prompt engineering writes the prompt; context engineering manages the whole evolving state. Avoid: prompt tuning. [Context Engineering]

System prompt
The always-loaded instructions defining the agent's behavior, tool use, and output format. Loaded first; in Claude Code it is ~4,200 tokens and invisible to the user. Should be written at the right altitude — specific enough to guide, general enough not to be brittle. [CC docs]

Compaction
Compressing a long history and large tool outputs into a structured, provenance-preserving summary, so a long-running agent can keep working without exhausting the window. Avoid: truncation (that just drops tokens; compaction summarizes). [Harnesses]

Just-in-time retrieval
Loading context on demand (the agent writes a targeted query and pulls only what it needs now) rather than pre-loading everything up front. The Vaadin MCP server is a JIT source of Vaadin docs. [Context Engineering]

Tools & extension

Tool
A defined capability the model can invoke to observe or act — read a file, run a command, query an API. Tool definitions live in the context window and deserve the same care as the main prompt; good tools are consolidated, namespaced, token-efficient, and return actionable errors. [Writing Tools]

MCP (Model Context Protocol)
An open standard for connecting agents to external tools and data. MCP servers expose tools, resources, and prompts. In Claude Code, MCP tool names load up front but schemas are deferred, loaded on demand to save context. Avoid: plugin, API (too generic). [CC docs]

Subagent
A specialized agent the lead agent delegates to, running in its own separate context window with a custom system prompt, specific tools, and independent permissions. It does side work and returns only a distilled summary (often ~1–2k tokens), preserving the lead agent's context. Avoid: thread, worker (unless in the orchestrator-workers pattern specifically). [CC docs, Multi-agent]

Skill
A packaged, reusable unit of agent guidance (instructions + optional scripts/resources) that auto-activates by context. Vaadin ships official skills (aura-theme, frontend-design, vaadin-form-layout). Avoid: command (that's a slash command), prompt. [vaadin/agent-skills]

Vaadin agent-readiness

Agent-ready (project)
A repo structured so an agent can work end-to-end with few mistakes: authoritative rule files, a doc-grounding MCP server, framework-aware skills, and a way for the agent to verify its own UI. The reference layout is vaadin/spec-driven-development-demo. [demo repo]

Vaadin MCP server
The official read-only doc-grounding server at mcp.vaadin.com (streamable-HTTP). Tools: search_vaadin_docs, get_full_document, get_vaadin_version. Framework-aware (Flow vs Hilla). Its job: stop the agent hallucinating outdated/invented Vaadin API. [announcement]

CLAUDE.md / AGENTS.md
Auto-loaded project context files for conventions, commands, and guardrails. AGENTS.md is the cross-tool-portable form; CLAUDE.md holds Claude-specific rules. Vaadin does not publish a single canonical CLAUDE.md to copy — its guidance ships as skills. [CC best practices]

Flow vs Hilla
Two Vaadin front-end models: Flow = UI written in Java (server-driven); Hilla = React front end with a typed Java back end. Agents frequently confuse the two — framework-awareness in tooling exists to prevent that.

Observability (Phoenix)

Trace
The complete path of a single request through the app — all spans sharing one trace_id. Avoid: log, session (a session groups multiple traces). [OpenInference]

Span
The atomic unit of work in a trace — one LLM call, one tool execution, one retrieval — carrying a name, parent span ID, timestamps, attributes, events, and status. Spans nest via parent_id. [OpenInference]

Span kind
The required openinference.span.kind attribute declaring what a span is. Values: LLM, EMBEDDING, CHAIN, RETRIEVER, RERANKER, TOOL, AGENT, GUARDRAIL, EVALUATOR, PROMPT. LLM = a model call; TOOL = an external tool call; CHAIN = glue between steps; AGENT = a span wrapping LLM+TOOL calls. [semantic conventions]

OpenTelemetry (OTel) & OpenInference
OTel is the vendor-neutral standard for emitting telemetry (the generic span/trace machinery). OpenInference is Arize's AI-specific layer on top — the span kinds and attribute names for LLM/agent observability. Phoenix consumes both; traces stay portable to any OTel backend. [OpenInference]

LLM-as-a-judge
Using an LLM to score or classify another system's output. In Phoenix, evaluators use function calling to return structured judgments, and return an explanation by default. Avoid: auto-grading, AI review. [Phoenix evals]

Dataset & Experiment (Phoenix)
A dataset is a versioned collection of examples (input + optional expected output). An experiment runs a task function over a dataset with evaluators attached, to compare versions of a prompt/model/agent side by side. This is the improvement loop. [Datasets & Experiments]

Path convergence
An agent-efficiency metric: minimum steps for this query type ÷ steps actually taken, a 0–1 score. Low = the agent wandered. Distinct from correctness (did it get the right answer at all). [Path convergence]

Learning-science terms (how we teach here)

Fluency strength vs storage strength
Fluency = in-the-moment retrieval (feels like mastery, fades fast). Storage = durable long-term retention (the real goal). Built by desirable difficulty: retrieval practice, spacing, and interleaving.

Zone of proximal development
The band of difficulty that is challenging but achievable with support — where each lesson aims to sit.
